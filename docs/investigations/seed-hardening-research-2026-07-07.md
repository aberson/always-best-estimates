## DIM: world-model-jepa

**Key findings**
- LeCun paper (local PDF, pp.24-27, sect 4.4-4.5): a JEPA is explicitly NON-generative — energy E = D(s_y, Pred(s_x, z)) is prediction error in REPRESENTATION space; it 'cannot easily be used to predict y from x'. So a raw JEPA emits neither returns nor sigma; both must be bolted on. Fig 12 caption explicitly permits the minimal variant: 'may use a fixed predictor with no latent' — a v1 can and should drop the latent z entirely.
- LeCun paper p.29 sect 4.5.2 ('Biasing a JEPA towards learning useful representations') directly endorses the needed mechanism: 'adding prediction heads that take s~_y as input and are trained to predict variables that are easily derived from the data and known to be relevant to the task.' This is the paper-sanctioned route from predicted latent -> expected return: a small supervised head g: s~_y -> H-day forward return, trained jointly with a small loss weight (plus a frozen-encoder linear probe as a diagnostic).
- Collapse prevention per the paper (pp.26-29): four criteria — maximize info in s_x, maximize info in s_y, make s_y predictable, minimize latent info. VICReg implements 1-2 via a per-dimension variance hinge + covariance-decorrelation computed over a batch ('dimension-contrastive', needs batch size comfortably larger than embedding dim). EMA target encoder + stop-gradient (BYOL/I-JEPA style) is the other standard route; a minimal design should use EMA-target + stop-grad AND VICReg variance/covariance terms on the online branch, because on tiny data either alone is fragile.
- LeCun paper sect 4.8 (pp.32-33) handles uncertainty via regularized latent variables sampled at plan time, producing multiple trajectories whose cost average+variance is optimized — i.e., even LeCun's own framework derives uncertainty from an ENSEMBLE of predictions, not from a single deterministic forward pass. This validates ensemble-based sigma for the v1.
- Data reality (verified inception dates): SPY 1993-01-22 (~8,400 daily bars), AGG 2003-09-22 (~5,700), ACWI 2008-03-26 (~4,600). The JOINT 3-asset panel is capped by ACWI at ~18.3 years ~= 4,600 rows — NOT 20-30 years. With H=21-day overlapping targets, that is only ~220 non-overlapping monthly outcomes per asset. Published OOS R^2 for monthly equity-return prediction is <1-2%; a JEPA should be expected to at best MATCH an EWMA/shrunk-historical-mean baseline. The plan's baseline-always-drives-the-pipeline decision is strongly validated.
- TS-JEPA literature exists and is young: 'Joint Embeddings Go Temporal' / TS-JEPA (arXiv 2509.25449, Ennadir et al. 2025) validates JEPA for time-series representation learning; MTS-JEPA (arXiv 2602.04643) notes that 'stabilizing latent-space self-distillation objectives on continuous, non-stationary time series remains practically challenging.' No established finance-specific JEPA baseline exists to copy — this component is research-grade, another reason it must be non-blocking.
- PyPortfolioOpt supports the target consumer contract: BlackLittermanModel(S, pi=..., absolute_views=..., omega='idzorek', view_confidences=<Kx1 array of confidences in [0,1]>) — so the WorldModel's (mu, sigma) must ultimately reduce to (absolute view returns in the prior's units) + (scalar confidence per view in [0,1]). Known edge: confidence 0 (or near-0) degenerates the Idzorek omega (divide-by-zero / huge omega, GitHub issue #95 lineage) — confidences must be clamped.
- Concrete minimal architecture that fits the data budget: context window L~=64 trading days x F features (AFML: frac-diff prices, realized vol, returns, lagged FRED macro) -> context encoder (1D-CNN or 2-layer GRU, ~10-100k params TOTAL) -> s_x in R^d with d=16-32; EMA target encoder (fixed momentum tau~=0.99, NOT an ImageNet-scale cosine schedule) encodes the future window t+1..t+H -> s_y; predictor = 1-2 layer MLP s_x -> s~_y; loss = D(s~_y, sg(s_y)) + VICReg variance/covariance terms; NO latent z in v1. Train per-asset with a shared encoder + asset embedding to triple effective sample count (~18-19k asset-days).

**Landmines**
- ISSUE: JEPA emits latents, not (mu, sigma) — the latent->return->confidence chain is unspecified and is the single highest-risk joint in the pipeline
  - WHY: Black-Litterman/Idzorek consumes per-asset expected returns + a [0,1] confidence. A deterministic latent predictor provides neither. If the plan hand-waves this, the build stalls or ships an uncalibrated head whose garbage confidences silently steer the portfolio.
  - ACTION: Specify the full chain as a named contract in the plan: (a) return head g: s~_y -> H-day log-return per asset, trained with Huber loss jointly with the JEPA loss at small weight (LeCun sect 4.5.2 sanctions this) + a frozen-encoder ridge probe as diagnostic; (b) sigma^2_total = across-ensemble variance of K=3-5 independently-seeded JEPA+head trainings + rolling walk-forward residual variance of the ensemble-mean forecast (skip heteroscedastic NLL heads and MC-dropout in v1 — both are unstable/miscalibrated at this data size); (c) calibration gate: empirical coverage of mu +/- 1.64*sigma must be ~90% on the walk-forward window, with a scalar recalibration factor if not; (d) Idzorek mapping c_i = |2*Phi(mu_i/sigma_i) - 1| clamped to [0.02, 0.95], so a no-information forecast -> c~=0 -> BL returns the market prior (graceful degradation by construction).
- ISSUE: Joint 3-asset history starts 2008-03 (ACWI inception), not 20-30 years back
  - WHY: yfinance/Stooq ETF data begins at fund inception; ACWI caps the aligned panel at ~4,600 rows. Training-set-size assumptions, walk-forward split dates, and 'decades of data' framing in the plan will all be wrong, and the first data-ingest step will surprise the builder.
  - ACTION: State the verified panel budget in the plan (SPY ~8.4k / AGG ~5.7k / ACWI ~4.6k daily bars; joint panel ~4.6k from 2008-03-26). v1 decision: train per-asset with a shared encoder + asset-ID embedding so SPY/AGG history before 2008 is still used; the aligned covariance/BL layer uses the joint window. Explicitly defer proxy backfill (e.g., regressing pre-2008 ACWI on SPY+EFA+EEM) to a labeled future step.
- ISSUE: Representation collapse will be silent and will produce plausible-looking constant forecasts
  - WHY: With ~4.6k samples a mis-weighted VICReg or EMA schedule collapses s~_y to a near-constant; the return head then emits near-identical mu for every asset/day, BL applies a constant tilt, and the UI shows a stable, confident, meaningless portfolio. No unit test catches this — it is the measurement-validity failure mode (the instrument lies).
  - ACTION: Mandate collapse instrumentation as a build-step deliverable, not an afterthought: log per-epoch (1) per-dimension std of s_x/s_y embeddings, (2) effective rank (RankMe-style), (3) variance of s~_y across a batch of distinct inputs; hard-fail training if per-dim std < threshold. Add the red-on-garbage anchor from .claude/rules/measurement-validity.md: a shuffled-target control run must score measurably WORSE than the real run before any JEPA forecast is trusted; assert score(real) > score(shuffled) in CI.
- ISSUE: Overlapping H-day targets make ordinary CV and naive backtests leak
  - WHY: With H=21, adjacent training targets share ~20/21 of their future window; random splits or unpurged folds leak labels and produce optimistic eval numbers that would wrongly promote JEPA over the baseline — exactly the decision this eval exists to gate.
  - ACTION: Pre-register the eval protocol in the plan: purged walk-forward (expanding-window) evaluation with an H-day embargo between train and test (Lopez de Prado AFML ch.7 — already the plan's feature-layer reference, so cite the same book for CV). The JEPA-vs-EWMA comparison metric (e.g., OOS MSE on H-day returns + sign accuracy + realized portfolio utility) is fixed BEFORE training; JEPA only replaces the baseline as the live view source after winning on this protocol.
- ISSUE: Bond ETF (AGG) price-only returns understate total return by ~the entire coupon (~4%/yr)
  - WHY: AGG's return is almost all distributions; unadjusted Close gives near-zero expected return, poisoning both training targets and the BL prior. Classic silent producer-consumer bug — everything runs, numbers are garbage.
  - ACTION: Plan's ingest step must specify adjusted close (yfinance auto_adjust=True; verify Stooq fallback adjustment parity or adjust from dividends explicitly) and log-returns computed from adjusted series ONLY. Add one ingest test asserting AGG's trailing-10y annualized mean return is materially > 0 (e.g., > 1%).
- ISSUE: Horizon/unit mismatch between the model's H-day mu and BL's prior-return units
  - WHY: PyPortfolioOpt's pi (market-implied prior) and S are conventionally annualized from daily data (x252); feeding H=21-day view returns as absolute_views against an annualized prior mis-scales every view by ~12x. This is the key/shape drift class from .claude/rules/code-quality.md.
  - ACTION: Define ONE source-of-truth constants module (HORIZON_DAYS, ANNUALIZATION=252, and to_annual(mu_H)/to_annual_sigma) imported by BOTH the WorldModel head and the BL layer; add an integration test that runs baseline-forecast -> BL -> optimizer end-to-end and asserts the view magnitudes land within sane annualized bounds (|view| < 50%/yr).
- ISSUE: Idzorek confidence edge cases crash or degenerate the BL layer
  - WHY: view_confidences at exactly 0 causes division-by-zero/degenerate omega in PyPortfolioOpt's idzorek path; confidence 1.0 forces omega->0 (view treated as certain), letting a garbage forecast dominate the prior.
  - ACTION: Clamp confidences to [0.02, 0.95] in the adapter between WorldModel and BL; unit-test both clamp edges plus the all-low-confidence case (result must approximately equal the market prior weights).
- ISSUE: FRED macro look-ahead leakage via revision/release lag
  - WHY: fredapi returns final revised series stamped at observation date; CPI/GDP/etc. were not knowable on that date. Features built naively give the model information from the future, inflating backtest results and misleading the JEPA-vs-baseline gate.
  - ACTION: Feature-layer step must lag every FRED series by a per-series declared release delay (e.g., CPI +15 calendar days, monthly employment +7) as an explicit column in a config table, and fit all scalers on training folds only (expanding-window). Add this to the eval-protocol pre-registration.
- ISSUE: Retraining inside the ~5-min refresh loop, and intraday partial bars polluting daily features
  - WHY: Daily-frequency forecasts only change once per completed trading day; retraining or re-featurizing on a live partial bar every 5 minutes burns compute and makes forecasts jitter intraday, destroying reproducibility of the displayed card.
  - ACTION: Plan separates cadences: a nightly (or weekly) TRAIN job producing a versioned checkpoint + frozen forecast row in SQLite, vs. the 5-min REFRESH loop that only re-reads prices for display and re-serves the cached forecast. Feature pipeline consumes completed daily bars only (drop today's bar until close).
- ISSUE: JEPA quality becoming a ship gate despite the plan's stated intent
  - WHY: With ~220 effective monthly samples per asset, parity-with-EWMA is the realistic best case and losing is likely; if any pipeline stage imports the JEPA directly instead of the WorldModel interface, a bad/absent model blocks the whole product.
  - ACTION: Make the WorldModel Protocol the FIRST build step: interface forecast(features) -> {asset: (mu_H, sigma_H)} with two registered implementations — EWMABaseline (always available, default) and JEPAModel (optional, behind the walk-forward win gate). Downstream stages (BL, optimizer, UI cards) integration-test against EWMABaseline only; the UI card for the world-model stage displays WHICH implementation produced the live forecast.
- ISSUE: EMA-target + VICReg hyperparameters tuned for ImageNet-scale runs fail at 4.6k samples
  - WHY: Momentum schedules that cosine-anneal tau->1.0 over hundreds of thousands of steps, or VICReg batch sizes < embedding dim, are the published defaults; copied blindly they collapse or never converge on this data. MTS-JEPA (2602.04643) confirms non-stationary-series self-distillation instability is the norm.
  - ACTION: Pin v1 hyperparameters in the plan: fixed tau=0.99, batch>=256 with embedding dim d<=32, VICReg (var,cov,inv) weights starting at (25,1,25)-family defaults scaled down, total params <100k, and NO latent z (fixed predictor per LeCun Fig 12 caption). Treat these as pinned-until-eval, changed only via the pre-registered eval.

**Recommendations**
- Architecture (write into the plan verbatim): x = last 64 trading days of AFML features per asset -> shared context encoder (2-layer GRU or 1D-CNN, <100k params, asset-ID embedding) -> s_x in R^32; EMA target encoder (tau=0.99, stop-gradient) on the future window t+1..t+H -> s_y; 2-layer MLP predictor s_x -> s~_y; loss = ||s~_y - sg(s_y)||^2 + VICReg variance-hinge + covariance terms; return head g(s~_y) -> H-day log-return trained jointly at 0.1x loss weight. H=21 trading days, single fixed horizon in v1.
- Sigma = deep ensemble (K=3-5 seeds) disagreement variance + rolling purged walk-forward residual variance; calibrate with an empirical-coverage check (mu +/- 1.64 sigma should cover ~90% of realized H-day returns) and a single scalar recalibration multiplier. Do NOT use MC-dropout or a Gaussian-NLL head in v1.
- Idzorek adapter: annualize (mu_H, sigma_H) via a single shared constants module; confidence c_i = |2*Phi(mu_i/sigma_i) - 1| clamped to [0.02, 0.95]; all-confidences-low must reproduce ~market-prior weights (unit-test this — it is the pipeline's graceful-degradation guarantee).
- Sequence build steps so the pipeline never depends on JEPA: Step order = data ingest -> feature layer -> WorldModel Protocol + EWMABaseline -> BL + optimizer + UI end-to-end on baseline -> JEPA as a later, independently-gated step. JEPA promotion to live requires winning the pre-registered purged walk-forward eval vs EWMA; until then the UI card shows 'baseline' as the active world model with JEPA metrics displayed alongside as shadow-mode output.
- Instrument collapse from day one: per-dim embedding std, effective rank, and prediction-variance-across-inputs logged every epoch, with hard training aborts on threshold breach, plus a shuffled-target control run asserted worse than the real run (red-on-garbage anchor).
- Set expectations in the plan's own prose: ~4.6k joint daily rows / ~220 effective monthly outcomes per asset means JEPA is a research showcase riding a production baseline, not the alpha engine; frame its UI card accordingly (show ensemble mu, sigma, and calibration coverage so the operator can see when it is uninformative).

**Open decisions**
- Horizon H: fix at 21 trading days (1 month) in v1, or condition the predictor on H? (Recommend fixed 21; horizon-conditioning is a v2 feature.)
- Ensemble size K and retrain cadence: K=3 vs 5; nightly vs weekly retrain job (forecast only changes once per trading day regardless).
- Return-head coupling: joint training at small loss weight (LeCun sect 4.5.2) vs frozen-encoder probe only — recommend joint + frozen ridge probe as diagnostic, but the plan must pick one as the mu source.
- Data-window policy: accept 2008-03 joint panel with per-asset pre-2008 pretraining (recommended), vs proxy-backfill ACWI before 2008 (defer), vs joint-only.
- Exact mu/sigma -> Idzorek confidence formula: sign-probability mapping |2*Phi(mu/sigma)-1| (recommended) vs variance-ratio shrinkage vs fixed confidence tiers — must be pinned before the BL step is built.
- Shadow-mode display: does the JEPA card render before the model passes the promotion gate (recommended: yes, labeled 'shadow — not driving allocation'), or is it hidden until promoted?
- VICReg-only vs EMA-only vs both for anti-collapse: recommend both (EMA+stop-grad primary, VICReg variance/covariance as insurance), but this doubles hyperparameters — plan should name the fallback (drop VICReg terms first) if training is unstable.

## DIM: scope-sequencing

**Key findings**
- V1 as stated bundles 5 independent risk classes: (a) flaky external data (yfinance is scrape-based and actively rate-limited per ranaroussi/yfinance discussion #2431 and 2025-2026 alternative-source articles), (b) research-grade ML (JEPA on financial time series is bleeding-edge: Fin-JEPA is a 2025/2026 SSRN preprint claiming to be the FIRST JEPA application to financial time series; MTS-JEPA/Phys-JEPA are 2026 arXiv papers), (c) quant methodology (AFML frac-diff/triple-barrier/sample-weights), (d) numerical optimization (BL+Idzorek+cvxpy with stateful turnover constraints), (e) a full-stack app (FastAPI+React+scheduler). Any one can stall the first end-to-end; skeleton-first is clearly correct.
- The pluggable WorldModel interface in the design is the exact lever for skeleton-first: an EWMA baseline implementing the SAME interface (per-asset mu + sigma over horizon H) exercises BL-Idzorek, cvxpy, storage, UI, and scheduler with zero ML risk. PyPortfolioOpt supports this today: BlackLittermanModel has the Idzorek method built in (omega='idzorek' + view_confidences), the package is declared 'mature' (stable, no major new features, bug fixes only), and its cookbook shows 1.5.6 running under Python 3.12 + numpy 2.3.4 — so the BL stage is NOT blocked on the model and can ship in the skeleton phase.
- Scope mismatch inside the AFML bundle: triple-barrier labeling + uniqueness sample-weights exist to train SUPERVISED CLASSIFIERS (meta-labeling). The V1 dataflow has no consumer for them — JEPA is self-supervised (predicts latent representations, not labels) and BL consumes (mu, sigma), not labels. Only frac-diff (FFD) is a genuine V1 feature-layer citizen (stationarity-preserving input transform for the model). Triple-barrier/sample-weights/purged-CV are orphan components in V1 and should be deferred to a V2 supervised/meta-label phase.
- JEPA does not natively emit (expected return, uncertainty): it emits latent embeddings via context-encoder -> predictor -> EMA target encoder. Getting (mu, sigma) requires an explicit readout head (e.g., Gaussian-NLL head) — an extra design decision the plan must name, or the build-step dev agent will improvise. Collapse prevention also needs more than EMA momentum (Fin-JEPA uses SIGReg; literature notes EMA alone does not guarantee non-collapse; VICReg-style variance terms are standard). Data is tiny (3 assets x ~15-25y daily ~= 4-6k rows/asset); Fin-JEPA's reference architecture is only 367K params.
- Idzorek consumes per-view confidences in [0,1]; the WorldModel emits sigma. The sigma->confidence mapping is an undesigned seam that BOTH the EWMA baseline and JEPA must feed. If the skeleton's baseline doesn't emit genuine uncertainty, the Idzorek path is never actually exercised until JEPA lands — defeating skeleton-first. The EWMA baseline must therefore derive sigma (e.g., trailing std of its own forecast errors) from day one.
- The turnover constraint makes the optimizer STATEFUL: it needs previous weights, which don't exist on first run and don't survive restarts unless allocations are persisted. This forces the SQLite schema (allocations/runs tables) into Phase 0 and a cold-start rule (skip/relax turnover on first run) into the optimizer step.
- Workspace pipeline constraints that shape the plan: plan.md must live at project root (or plans/docs/) with scrapable '## 1. What This Is' goal + '### Step N:' units + '**Status:** DONE' markers, each step needs Problem/Type/Issue/Flags fields, Automated vs Manual (M1/M2...) sections must be split, Type: operator steps must not produce code, soak/observation steps must be 'Type: wait' (build-phase halt contract), default '--reviewers code' with 'full' only for the real UI step (requires --start-cmd + --url), and Phase 0 must run 'observatory register always-best-estimates --owned' + check the chosen ports against 'observatory ports' (Vite on Windows needs server.port + strictPort + host 127.0.0.1 per workspace memory).
- The 5-minute refresh against DAILY price data means most scheduled runs are recompute-no-change. The scheduler must separate 'fetch new data at most ~once/day per source' from 'recompute pipeline from cache every 5 min or on demand', or it will hammer the already-rate-limited yfinance and get the IP blocked mid-development.
- Local PDFs papers/Idzorek_onBL.pdf and papers/10356_a_path_towards_autonomous_mach.pdf (LeCun JEPA position paper) exist in the workspace and should be cited as references in the plan's JEPA and BL steps; they were not readable in this subagent environment (pdftoppm missing) — facts above verified via web instead.

**Landmines**
- ISSUE: JEPA-before-harness ordering: if the plan builds the JEPA (or the full AFML layer) before a working downstream, an unproven research model blocks BL/optimizer/UI indefinitely and the first end-to-end is weeks out.
  - WHY: JEPA on financial time series has essentially no established recipe (Fin-JEPA is a first-of-kind 2025/26 preprint); collapse and no-signal outcomes are likely on 4-6k rows/asset. Every downstream stage would inherit that uncertainty.
  - ACTION: Order phases so the EWMA-baseline skeleton (ingest -> EWMA WorldModel -> BL-Idzorek -> cvxpy -> UI cards) is a completed, tested phase BEFORE any JEPA step. Freeze the WorldModel interface in the skeleton phase with a contract test (emit per-asset mu, sigma, horizon H) that every implementation must pass; JEPA later slots in behind a config toggle.
- ISSUE: Triple-barrier labeling + sample weights (and purged CV) have NO consumer in the V1 dataflow — JEPA is self-supervised and BL consumes (mu, sigma), not classification labels.
  - WHY: They are AFML machinery for supervised meta-labeling. Building them in V1 is orphan code: untested-in-anger, maintenance load, and a false sense of 'AFML done'. Plan-review should reject components no step consumes.
  - ACTION: Move triple-barrier, meta-labeling, sample-weights, and purged/embargoed CV to an explicit 'Deferred to V2' section of plan.md with the sentence 'no V1 consumer; becomes relevant when a supervised confidence/meta-label head is added'. Keep only frac-diff (FFD) + returns/vol/macro alignment in the V1 feature phase.
- ISSUE: sigma -> Idzorek-confidence mapping is an undesigned seam; if the EWMA baseline emits only mu (no uncertainty), the Idzorek path is dead code until JEPA arrives.
  - WHY: Skeleton-first only de-risks stages it actually exercises. PyPortfolioOpt's idzorek path needs per-view confidences in [0,1]; nothing in the design says how sigma becomes confidence.
  - ACTION: The skeleton phase must include one step that (a) defines the mapping function ONCE in a leaf module (one source of truth per code-quality rule) with unit tests at boundary values (sigma->0 => conf->1, sigma large => conf->0), and (b) requires the EWMA baseline to emit genuine sigma (trailing std of its own forecast errors), asserted by the WorldModel contract test.
- ISSUE: Turnover constraint is stateful — first run has no previous weights, and restarts lose in-memory state.
  - WHY: cvxpy turnover terms need w_prev; an unhandled cold start crashes or silently drops the constraint, and un-persisted allocations make every restart a cold start.
  - ACTION: Put allocations/runs tables in the Phase 0 SQLite schema; the optimizer step's Done-when must include 'cold start (no prior allocation row) handled by skipping/relaxing turnover, covered by a test; second run reads w_prev from SQLite, covered by an integration test through the FastAPI route'.
- ISSUE: yfinance rate limiting will bite twice: during iterative development (repeated fetches) and from the 5-min scheduler (hammering a scrape-based endpoint).
  - WHY: yfinance is unofficial scraping; Yahoo actively rate-limits/temp-bans (github discussion #2431, multiple 2025-26 reports). A blocked IP mid-build-phase stalls autonomous runs.
  - ACTION: Phase 1 (ingest) must be cache-first: idempotent upsert into SQLite, backfill once, all downstream dev reads from cache; exit criterion includes an offline test (network-off, pipeline still completes from cache). Phase 3 (scheduler) must specify fetch-at-most-once-per-day-per-source vs recompute-every-5-min-from-cache as separate code paths.
- ISSUE: Building the full 3-source fallback chain (yfinance + Stooq + Google Finance scrape) before the pipeline exists is breadth-before-depth waste; the Google scrape is best-effort by its own description.
  - WHY: Multi-source reconciliation (discrepancy checks, symbol mapping, corporate-action differences) is real work that delivers zero user-visible value until a pipeline consumes the data.
  - ACTION: Phase 1 defines a PriceSource protocol but implements ONLY yfinance. Stooq fallback is its own later phase/step with a simulated-primary-failure exit test. Google Finance scrape is marked explicitly cuttable/stretch in plan.md (or deferred to V2) so plan-review doesn't treat it as committed scope.
- ISSUE: JEPA acceptance without a pre-registered baseline gate — 'JEPA is integrated' quietly becomes 'JEPA is the default forecaster' with no evidence it beats EWMA.
  - WHY: Workspace measurement-validity rule exists because a transcript-scored bake-off previously shipped a 100%-no-diff coder. The analogous failure here: judging the JEPA by training-loss curves or vibes instead of production-path forecasts.
  - ACTION: The JEPA phase exit criterion must be pre-registered in the plan: walk-forward evaluation of (mu, sigma) produced through the production WorldModel interface, compared against the EWMA baseline on the same windows (e.g., forecast MSE/IC + sigma calibration), plus a collapse anchor check (latent variance above threshold; garbage/constant model must score worse). JEPA ships toggle-OFF if it loses; V1 'done' = 'pipeline correct + JEPA honestly evaluated behind toggle', NOT 'JEPA wins'.
- ISSUE: Plan step for the JEPA that just says 'emit expected return + uncertainty' leaves the readout-head and anti-collapse mechanisms unnamed.
  - WHY: JEPA natively outputs latent embeddings; (mu, sigma) requires an explicit head (e.g., Gaussian-NLL readout) and collapse prevention needs more than EMA (VICReg-style variance term or similar). An unnamed mechanism means the build-step dev agent improvises a design decision mid-build.
  - ACTION: The JEPA step's Problem/Produces must name: context encoder + EMA target encoder + predictor, the specific anti-collapse regularizer, the specific (mu, sigma) readout mechanism, and a parameter budget sized to the tiny dataset (order 100K-500K params, citing Fin-JEPA's 367K as the reference point). Cite papers/10356_a_path_towards_autonomous_mach.pdf in the step.
- ISSUE: Lookahead bias in the FRED macro join: FRED series are published with a lag and revised; joining on observation date instead of availability date trains the model on future information.
  - WHY: Discovered late, this invalidates every trained model and every evaluation — the most expensive possible rework point. It is cheap to prevent at feature-layer time and near-impossible to retrofit.
  - ACTION: The feature-layer phase must include an as-of/release-lag alignment step (at minimum a per-series publication-lag shift; ALFRED vintages deferred to V2) and a mandatory no-lookahead unit test: feature value at date t computed only from data available at t. Make this a named exit criterion, not a code comment.
- ISSUE: Scheduler soak and UI verification steps will violate the build-phase halt contract if typed wrong (mid-run operator halts are defects).
  - WHY: Workspace rule: only wait-type steps, quality-gate fails, stop-and-audit, predicate errors, and merge conflicts may halt /build-phase; UI 'operator looks at it' checks mid-phase are a known defect pattern.
  - ACTION: Mark the scheduler soak ('leave running N hours, ledger shows periodic runs') as 'Type: wait'. ## DIM: optimization-and-asset-overlap

**Key findings**
- SPY-ACWI overlap is confirmed and large: US equities are ~60-64% of MSCI ACWI (base index 62.57% as of Dec 2023; ACWI Screened variant 63.85% as of May 2026 per MSCI factsheet), and SPY/ACWI return correlation is ~0.94-0.96 over the last 15 years. The 3-asset covariance matrix therefore has one near-zero eigenvalue along the (SPY - beta*ACWI) direction: the equity 2x2 block is near-singular.
- Consequence for MVO: mean-variance is an error maximizer (Michaud 1989, Best-Grauer 1991). With rho~0.95, the optimizer sees SPY-vs-ACWI as an almost-free swap, so basis-point-level differences in the model's mu estimates produce bang-bang allocation: the equity sleeve flips ~100% between SPY and ACWI as mu jitters between refreshes. Long-only (w>=0) acts as implicit shrinkage (Jagannathan-Ma 2003) and prevents explosive long-short positions, but does NOT prevent the discontinuous SPY<->ACWI flip; a turnover constraint then binds every rebalance and produces slow path-dependent drift instead of a stable answer.
- Black-Litterman prior landmine specific to this menu: pi = delta * Sigma * w_mkt requires market-cap weights, but SPY's constituents are a subset of ACWI's, so any cap-derived w_mkt double-counts US large-cap. Also ETF AUM is NOT asset-class market cap (wrong unit). The plan must define w_mkt as an explicit, documented vector over the 3 tickers rather than fetching caps.
- Ledoit-Wolf is the right shrinkage here and is available off the shelf: sklearn.covariance.LedoitWolf or PyPortfolioOpt risk_models.CovarianceShrinkage(prices).ledoit_wolf(). With N=3 and T~1000+ daily observations the motive is NOT estimation scarcity (classic LW motivation) but conditioning: shrinking toward a scaled-identity / constant-correlation target lifts the smallest eigenvalue off zero, tames the near-arbitrage direction, and guarantees a strictly PSD matrix for cvxpy. Note: AFML/Lopez-de-Prado Marchenko-Pastur denoising is a large-N random-matrix technique and is meaningless at N=3 -- the plan should explicitly say LW yes, RMT denoising no, despite the AFML framing of the project.
- cvxpy DPP facts (confirmed against cvxpy DPP docs): cp.quad_form(w, P) with P a Parameter is NOT DPP. The documented workaround is S_sqrt = cp.Parameter((n,n)); risk = cp.sum_squares(S_sqrt @ w); S_sqrt.value = scipy.linalg.sqrtm(S) (or Cholesky factor transpose) computed outside the DSL. Additionally, gamma as a separate Parameter multiplying sum_squares(...) breaks DPP (parameter-times-parameter-expression); fold sqrt(gamma/2) into S_sqrt.value, or keep gamma a plain Python float. Assert prob.is_dcp(dpp=True) in a unit test if the parameterized path is chosen.
- cvxpy PSD-check hazard (confirmed via cvxpy issues #1424/#1421 and PyPortfolioOpt issue #472): quad_form runs an ARPACK smallest-eigenvalue PSD check that is known to fail with ArpackNoConvergence precisely on ill-conditioned/near-singular covariance matrices -- i.e., exactly the matrix this asset menu produces without shrinkage. Mitigations: quad_form(w, S, assume_PSD=True) or cp.psd_wrap(S), used ONLY after PSD is enforced upstream (LW shrinkage + symmetrize (S+S.T)/2 + eigenvalue clip at >=1e-10), or avoid quad_form entirely via the sum_squares(chol.T @ w) form.
- Concrete recommended formulation (3 assets): w = cp.Variable(3); objective cp.Maximize(mu @ w - 0.5 * gamma * cp.sum_squares(S_sqrt @ w)); constraints [cp.sum(w) == 1, w >= 0, w <= w_max, cp.norm1(w - w_prev) <= tau]. All pieces are convex/QP-representable; norm1 turnover is exact (no need for buy/sell split variables at this scale). Solve with an explicitly pinned solver (Clarabel or OSQP) -- cvxpy's default solver choice has changed across versions -- and check prob.status in {OPTIMAL, OPTIMAL_INACCURATE}, treating INACCURATE as a surfaced warning; clip w < 1e-8 to 0 and renormalize before persisting/display.
- Numerical scale: daily-return covariances are O(1e-4..1e-5), making objective terms tiny relative to solver tolerances. Annualize both mu and S (x252) so terms are O(1e-2..1e-1). Units mismatch (annualized BL posterior mu vs daily S) is a classic silent bug: weights come out plausible-looking but gamma is effectively off by ~252x.
- Risk-aversion lambda: canonical choices are He-Litterman delta = 2.5, typical range 2-4; PyPortfolioOpt provides black_litterman.market_implied_risk_aversion(market_prices) = (R - R_f)/sigma^2 (annualized, returns a float). Confirmed hazard: in a drawdown window the market-implied value can be near-zero or NEGATIVE, and a negative gamma flips the QP's concavity (cvxpy DCP error or garbage weights). It must be clamped. Consistency requirement: the SAME delta should parameterize the BL prior (pi = delta*Sigma*w_mkt) and the MVO objective; if they diverge, the optimizer deviates from the benchmark even with zero views, destroying BL's main calibration property.
- PyPortfolioOpt Idzorek implementation (confirmed from pypfopt.black_litterman source): omega='idzorek' with view_confidences (Kx1, each in [0,1]) required, else ValueError. Edge cases: conf == 0 is special-cased to omega = 1e6 (view effectively ignored); conf == 1 gives alpha = (1-conf)/conf = 0 hence omega = 0 (view treated as CERTAIN), which combined with a near-singular Sigma produces extreme posterior tilts. Model-emitted confidences must be clamped away from the endpoints.
- PyPortfolioOpt packaging status (confirmed): the original robertmartin8 release (1.5.5) pins numpy<2.0.0 (GitHub issue #605), which conflicts with a modern 2026 Python 3.12 stack. The project has moved to the PyPortfolio GitHub org, whose releases page shows v1.6.0 with newer Python/pandas support. The plan must pin the maintained v1.6.0+ line and smoke-test the import against the chosen numpy at env-setup time, with a hand-rolled BL posterior (~30 lines of documented linear algebra) as the named fallback if dependency resolution fails.
- Turnover semantics need definition for a ~5-min refresh loop over daily data: mu/S legitimately change ~once/day (plus on-demand view changes), so 'previous weights' must mean the last PERSISTED allocation in SQLite, not the previous solver output, and the pipeline should skip re-solving when the input hash (mu, S, w_prev, constraints) is unchanged -- otherwise solver jitter plus a binding turnover constraint manufactures spurious weight drift every 5 minutes.

- ISSUE: Turnover constraint infeasibility and undefined baseline: cp.norm1(w - w_prev) <= tau is infeasible when w_prev is absent (first run), doesn't sum to 1 (drift/renormalization), or violates the box; and at a 5-min refresh the meaning of 'previous weights' is ambiguous.
  - WHY: First production run or a resumed run after a DB wipe hits an infeasible problem and the stage errors; a binding turnover constraint re-applied every 5 minutes against the last solver output manufactures path-dependent weight drift from pure solver jitter.
  - ACTION: Plan the guard explicitly: drop the turnover constraint when no persisted weights exist; renormalize persisted w_prev to the simplex before use; on INFEASIBLE status retry once without the turnover constraint and flag the relaxation in the stage card; define turnover baseline = last persisted allocation in SQLite; skip re-solving entirely when the input hash (mu, S, w_prev) is unchanged.

  - WHY: An unpinned solver makes results non-reproducible across environment rebuilds; unhandled INACCURATE status or raw negative weights leak into the UI/persistence as confusing artifacts.

- ISSUE: TARGET MISMATCH (required plan decision): triple-barrier emits classification labels {-1,0,+1}, but the WorldModel contract is continuous per-asset (expected return, uncertainty) over fixed horizon H, and BL/Idzorek consumes Q in return units + confidence in [0,1].
  - WHY: If the plan naively wires 'AFML labeling' into the forecast path, the model trains on class labels that cannot produce a Q vector; converting class probabilities to E[r] (P(+1)*PT - P(-1)*SL) loses magnitude and horizon consistency and is not defensible for BL views.
  - ACTION: Write an explicit decision record in plan.md: V1 forecast head regresses fixed-H forward log-return per asset with a heteroscedastic head (predict mu and sigma, Gaussian NLL) or a small ensemble for sigma; triple-barrier labeling is EXCLUDED from the forecast path and kept only as (a) an offline evaluation harness (hit-rate of sign(mu_hat) vs barrier label) and (b) a deferred Phase-2 meta-labeling stage whose P(primary-signal-correct) maps naturally onto Idzorek confidence. Forbid feeding {-1,0,+1} into BL Q anywhere.
- ISSUE: The seemingly-clever middle option — regress getBins' `ret` at barrier-touch time t1 — produces VARIABLE-horizon targets (t1 differs per event depending on which barrier hits first).
  - WHY: BL views and the covariance matrix are defined at one fixed horizon; mixing variable-horizon returns into Q silently breaks the units of the blend and the utility optimization.
  - ACTION: Plan must state the V1 target as vertical-barrier-only, i.e. plain H-bar forward log-return (which is what triple-barrier degenerates to with only the time barrier), and name variable-horizon regression as a rejected alternative.
- ISSUE: Depending on or vendoring the charlesrambo repo directly.
  - WHY: No license file (legally unvendorable), not pip-installable, Chapter_7 uses removed pd.Series.append and sklearn private _BaseKFold (crashes on pandas>=2.0), Chapter_3 is coupled to the repo's own Chapter_20 multiprocessing, and the author tags code 'minimally debugged'.
  - ACTION: Treat the repo as a readable reference only. Plan a small in-project package (e.g. backend/afml/: fracdiff.py, labeling.py, purged_cv.py, ~60-150 lines each) implemented against current pandas/sklearn public APIs, each with anchor tests. If a dependency is preferred for CV, use skfolio's CombinatorialPurgedCV instead — never import the GitHub scripts.
- ISSUE: No minimal-d ADF search exists in the reference repo (or in most snippets people copy).
  - WHY: The plan says 'choose d via ADF' but the referenced code only plots a d-sweep; without a specified algorithm the builder will improvise, and improvised versions often pick d on the FULL history including test periods (leakage) or use inconsistent pass criteria.
  - ACTION: Specify the algorithm in the plan: grid d in {0.0,0.05,...,1.0}, fixed-width FFD at chosen tau, adfuller(regression='c'), select smallest d with p<0.05 (fallback d=1.0 if none passes), computed on TRAINING data only; log {d, adf_p, corr_with_raw, window_len} per series.
- ISSUE: Re-estimating d (and FFD weights) on every ~5-min refresh.
  - WHY: d is a model hyperparameter: re-fitting it continuously changes the feature definition under the live model, creates backtest-vs-live drift, and constitutes lookahead in any historical evaluation. Also pointless for daily bars that change once per day.
  - ACTION: Freeze d at (re)train time: persist {d, tau, window_len, adf_p, corr} in SQLite model metadata; the inference path loads frozen weights and only applies them to the latest window; d is recomputed exclusively inside the scheduled retrain job.
- ISSUE: FFD burn-in vs short history: repo default threshold=1e-5 yields weight windows that can exceed 1,000 daily bars for typical d~0.3.
  - WHY: The joint panel is capped by ACWI (inception 2008-03-26, ~4,600 bars); a 1,000+ bar burn-in silently discards ~25% of usable history and shrinks an already-small effective sample (~N/H independent samples).
  - ACTION: Plan sets tau=1e-4 or an explicit max window (e.g. 252 bars) as the V1 default, asserts post-burn-in sample count >= a named floor at pipeline startup (fail loud, per the workspace fail-loud rule), and surfaces {d, window, n_samples} on the feature-stage UI card.
- ISSUE: Naive KFold / random split (or default sklearn CV inside hyperparameter search) on overlapping H-bar labels.
  - WHY: Adjacent samples share (H-1)/H of the return path; standard splits leak test-label information into training and report fantasy skill, which then flows into BL as overconfident Q — the exact 'instrument lies' failure class in the workspace measurement-validity rule.
  - ACTION: Plan mandates purged + embargoed time-ordered splits for ALL model selection and evaluation, embargo >= H bars; explicitly ban shuffle/random splits in the training code; include a leakage unit test asserting no training label interval overlaps any test interval and the embargo gap holds.
- ISSUE: JEPA validation split leakage via overlapping windows (separate from label overlap).
  - WHY: JEPA train/val pairs are (context window, target window); windows sampled near the train/val boundary overlap across the split, making val loss meaningless even though no supervised labels are involved.
  - ACTION: Plan specifies a single chronological train/val cut for JEPA pretraining with a gap >= (context_len + target_len) bars between the last train window end and first val window start — same purge/embargo logic applied to self-supervised windows.
- ISSUE: Live retrain lookahead: the most recent H bars have incomplete labels (t1 in the future) at every refit.
  - WHY: An always-on engine that periodically refits and naively builds labels up to 'now' trains on partially-formed forward returns (or NaNs), a silent lookahead/garbage-label bug that no unit test of the labeler alone catches.
  - ACTION: Retrain job must drop all samples with t1 > last available bar; add a test asserting max(train_t1) <= dataset_end; document that the newest H bars are inference-only.
- ISSUE: Over-engineering the uniqueness machinery in V1 (sequential bootstrap, indicator matrix).
  - WHY: The repo's indicator matrix is O(events x timestamps) memory and sequential bootstrap is slow; the repo also lacks return-attribution and time-decay weights entirely, so 'use Chapter 4' under-delivers vs the book. For 3 assets of daily data the marginal value over purged CV is small.
  - ACTION: Scope V1 as: purged+embargoed splits MANDATORY; average-uniqueness per-sample loss weights (computable in O(N) from rolling concurrency counts = for fixed-H labels simply 1/#overlapping) OPTIONAL; sequential bootstrap and time-decay weights explicitly OUT of V1, listed as deferred.
- ISSUE: Horizon-unit drift: H appears in >=5 places (label/forward-return construction, JEPA target window, BL view horizon, covariance annualization, embargo length) and the reference code mixes calendar-day Timedeltas with trading-bar series.
  - WHY: Duplicate H definitions WILL drift (workspace code-quality one-source-of-truth rule); calendar-vs-trading-day mismatch alone changes an intended 21-bar horizon into ~15 bars of actual data.
  - ACTION: Plan declares one HORIZON_BARS constant (trading bars, e.g. 21) in a leaf module imported by the labeler, model, BL blender, and CV; regression test asserts consumers import the same object (assert 'is').
- ISSUE: Applying FFD to FRED macro series, and aligning FRED by observation date.
  - WHY: Monthly macro series have only ~220 obs since 2008 — FFD burn-in destroys them; separately, fredapi returns latest-REVISED values and releases lag their reference period, so observation-date alignment leaks future information into historical features.
  - ACTION: V1 restricts FFD to daily price series; macro features use simple stationary transforms (YoY change, first difference, z-score) and are shifted by a conservative per-series publication lag in the feature matrix; note ALFRED vintage data as a deferred correctness upgrade. Coordinate with the data-ingestion dimension so this lands once.
- ISSUE: No garbage-fails-low anchors for the feature layer.
  - WHY: Per the workspace measurement-validity rule, an instrument that can't fail garbage can't be trusted; the d-search and purged CV are exactly the kind of code that silently returns plausible numbers when broken.
  - ACTION: Plan includes anchor tests: min-d search returns d=0 (or near) for white noise and d~1 for a simulated random walk; FFD weights match a hand-computed small case; purged-CV leakage test from the landmine above; feature pipeline aborts (not warns) when any input series is shorter than window_len + floor.

**Recommendations**
- Define the V1 AFML scope table in plan.md explicitly — IN: fixed-width FFD on log-prices with train-time min-d ADF search; purged + embargoed chronological splits (single test path is enough for V1; combinatorial CV optional later); optional average-uniqueness loss weights. ## DIM: runtime-architecture

**Key findings**
- APScheduler stable is 3.11.3 (2026-06-28); 4.x is alpha-only (latest 4.0.0a6, 2025-04-27) - verified on PyPI 2026-07-07. A plan that assumes 'APScheduler 4' is planning on vaporware; if APScheduler is used at all, pin 3.11.x.
- Workspace has ZERO existing APScheduler usage (grep across c:/Users/abero/dev = no hits). The one existing FastAPI lifespan pattern is brickomancer/src/brickomancer/main.py: @asynccontextmanager lifespan + await asyncio.to_thread(...) for blocking init - the local convention to mirror.
- Verified port map via 'uv run --project dev-observatory observatory ports': taken = 3000, 4000, 5173, 8000, 8080, 8502, 8503, 8600, 8765. Vite's default 5173 is ALREADY TAKEN (brickomancer) and two accidental collisions (3000, 8000) already exist - defaults are not safe; ports must be chosen and declared.
- For one job every ~5 min + on-demand in one process, a plain asyncio background task in FastAPI lifespan beats APScheduler and cron: 'while True: try: await asyncio.wait_for(trigger_event.wait(), timeout=300) except TimeoutError: pass; await run_pipeline()' gives timer + on-demand trigger (event.set() from POST handler) + coalescing + fixed-delay semantics in ~20 lines, zero deps. External cron does not exist on Windows 11 Home (Task Scheduler only) and would cold-start PyTorch weights every 5 minutes.
- CPU-bound PyTorch inference must leave the event loop: run the pipeline body via loop.run_in_executor with a dedicated ThreadPoolExecutor(max_workers=1). PyTorch releases the GIL in native ops so a thread suffices; a ProcessPoolExecutor would force model-weight reload per run or a persistent worker protocol - unjustified for V1. max_workers=1 doubles as the single-flight guarantee. Set torch.set_num_threads() and use torch.inference_mode() for forecast passes.
- uvicorn must run workers=1 (default) and WITHOUT --reload: N workers = N schedulers = duplicate runs + SQLite write contention; --reload on Windows drops session PATH via WatchFiles (codified workspace memory feedback_uvicorn_reload_path; brickomancer CLAUDE.md documents the same 'no --reload' rule). The exclusive port bind is the natural single-instance mutex - a second accidental launch fails fast on bind.
- SQLite WAL is the right mode here: PRAGMA journal_mode=WAL (persistent, set once), synchronous=NORMAL, busy_timeout=5000, foreign_keys=ON. WAL lets UI read endpoints run concurrently with the single pipeline writer, but SQLite still allows only ONE writer at a time - so all writes must flow through the pipeline thread's one connection. Long-lived reader connections block WAL checkpointing (WAL file grows unbounded); use short-lived per-request read connections and run wal_checkpoint(TRUNCATE) at end of each pipeline run.
- Data cadence mismatch is a runtime fact: SPY/ACWI/AGG bars and FRED series are DAILY, but the loop runs every 5 min. ~99% of runs will see unchanged inputs. The scheduler loop therefore needs a cheap freshness check that short-circuits to a 'skipped' run_log outcome (and a --force path on the manual trigger), or it hammers yfinance/FRED toward rate limits and burns CPU re-training/re-inferring on identical inputs.
- Schema that supports 'UI reads history, not just latest': runs(run_id PK, started_at_utc, finished_at_utc, status CHECK IN (queued,running,ok,error,skipped), trigger IN (schedule,manual,startup), error_text) + run_stages(run_id, stage, status, started_at_utc, finished_at_utc, detail_json) which directly powers the one-card-per-stage UI; prices(asset, date, open, high, low, close, adj_close, volume, source, fetched_at_utc, PK(asset,date)) upserted with provenance (yfinance/stooq/google); macro(series_id, date, value, fetched_at_utc, PK(series_id,date)); features(run_id, asof_utc, asset, name, value); forecasts(run_id, asof_utc, asset, horizon_days, mu, sigma, model_version); bl_posteriors(run_id, asof_utc, asset, prior_mu, view_mu, view_confidence, posterior_mu, posterior_sigma, detail_json for matrices); target_weights(run_id, asof_utc, asset, weight, prev_weight, turnover). 'Latest' = MAX(run_id) WHERE status='ok'. All timestamps UTC ISO 8601 (existing workspace convention); price/macro dates stay as trading-date strings, never localized.
- Frontend: two-mode split resolves 'always-on' cleanly - production mode is 'vite build' output served by FastAPI StaticFiles so exactly ONE process must stay up; the Vite dev server (pinned port, strictPort:true, host:'127.0.0.1' per feedback_vite_dev_port / feedback_vite_ipv6_only_on_windows - Vite binds ::1-only on Windows otherwise) is development-only, with server.proxy {'/api': 'http://127.0.0.1:<backend>'} to avoid CORS entirely. Polling beats streaming at this cadence: GET /api/runs/latest (+ /api/runs/{id}/stages, /api/history?limit=N) every 5-10s with a document.visibilitychange pause; SSE is an optional later upgrade for live in-run stage progress; WebSockets are unjustified.
- Workspace-convention hooks verified: register command is 'uv run --project dev-observatory observatory register always-best-estimates --owned --path <rel-path>' (until registered the project is treated owned=false/write-blocked); plan must live at plan.md or master_plan.md in root or plans//docs//documentation/ with scrapable '### Step N:' + '**Status:** DONE' units; CLAUDE.md needs a '## Commands' heading with runner-first-token commands and scrapable port shapes (127.0.0.1:PORT) for the observatory deriver; 'observatory ports' is the collision linter to run after declaring.

**Recommendations**
- Scheduler: plain asyncio background task created in FastAPI lifespan (asyncio.create_task on startup, task.cancel() + await on shutdown). Loop = wait on an asyncio.Event with timeout=300 (fixed-delay), then run the pipeline under an asyncio.Lock. POST /api/runs/trigger sets the event and returns 202 + run_id immediately. No APScheduler dependency needed for one job; if the team insists, apscheduler>=3.11,<4 with AsyncIOScheduler, coalesce=True, max_instances=1, misfire_grace_time=None.
- Concurrency shape: uvicorn single-process single-worker on 127.0.0.1; one dedicated ThreadPoolExecutor(max_workers=1) executes the synchronous pipeline (fetch -> features -> JEPA -> BL -> cvxpy -> persist) off the event loop; read endpoints are sync 'def' functions with short-lived read-only sqlite connections (file:...?mode=ro URI). This gives strict serialization of writes and a responsive UI during runs with zero locking cleverness.
- SQLite: one storage.py module owns connection creation and PRAGMAs (WAL, synchronous=NORMAL, busy_timeout=5000, foreign_keys=ON), the insert-coercion boundary (float()/int()/.item()), and the schema DDL - one source of truth per code-quality rule. Wrap each pipeline stage's writes in an explicit transaction keyed by run_id so a crash leaves an inspectable partial run with status='error', never orphan rows without a runs entry. wal_checkpoint(TRUNCATE) after each run.
- Schema: adopt the runs + run_stages + prices + macro + features + forecasts + bl_posteriors + target_weights layout from key_findings; every derived table carries (run_id, asof_utc, asset) so the UI can render both latest and history; run_stages IS the API for the one-card-per-stage UI (stage name, status, duration, one-line detail). Add indexes on runs(status, run_id) and each table's (asset, asof_utc). Include a retention note: at 5-min cadence with skip-gating, row volume is dominated by real daily runs (~1-5/day), so no pruning needed in V1 - state that explicitly so nobody builds pruning prematurely.
- Frontend: React+Vite dev server pinned (port from the unused range, strictPort, host 127.0.0.1, /api proxy); production/always-on mode is 'npm run build' + FastAPI StaticFiles mount so one process serves everything. - Ports: pick two currently-unused ports (e.g. backend 8140, Vite dev 5174 - both absent from the live map), declare both in CLAUDE.md '## Commands' in scrapable 127.0.0.1:<port> form, then run 'uv run --project dev-observatory observatory ports' to confirm no collision. ## DIM: black-litterman-idzorek

**Key findings**
- Idzorek paper (local PDF, pp.3-5): equilibrium prior is Pi = delta * Sigma * w_mkt (EXCESS returns, Formula 1); delta = (E(r_mkt) - r_f) / sigma^2_mkt (Grinold-Kahn implied risk aversion, ~3.07 in the paper, footnote 5); unconstrained weights recover as w = (delta*Sigma)^-1 * mu (Formula 2).
- Idzorek paper (pp.6-13): full BL formula E[R] = [(tau*Sigma)^-1 + P' Omega^-1 P]^-1 [(tau*Sigma)^-1 Pi + P' Omega^-1 Q]; P is KxN (for K absolute views on 3 assets P = I_3), Q is Kx1, Omega is KxK diagonal; tau guidance is contradictory in the literature (Lee 0.01-0.05, Satchell-Scowcroft 1.0, Blamont-Firoozye ~1/T) but tau cancels whenever omega is set proportional to tau (He-Litterman omega/tau = p_k Sigma p_k', or Idzorek mode).
- Idzorek confidence method exactly (paper Section 3.2, Formulas 12-19): per view k in isolation, (1) compute E[R]_k,100% under Omega=0, (2) w_k,100% = (delta*Sigma)^-1 E[R]_k,100%, (3) D_k,100% = w_k,100% - w_mkt, (4) target tilt = D_k,100% * C_k with user confidence C_k in [0,1], (5) solve min over omega_k of sum((w_k,% - w_k(omega_k))^2) s.t. omega_k > 0, (6) assemble diagonal Omega and run the full BL formula. Paper's explicit note (Formula 12 footnote): if a view is stated as TOTAL return, subtract the risk-free rate from Q_k first.
- PyPortfolioOpt CONFIRMED: pypfopt.black_litterman.BlackLittermanModel(cov_matrix, pi=None|'market'|'equal'|Series, absolute_views=dict|Series, Q=, P=, omega=None|'default'|'idzorek'|ndarray, view_confidences=Kx1 in [0,1], tau=0.05, risk_aversion=1, **kwargs incl. market_caps, risk_free_rate). Methods: bl_returns() -> pd.Series posterior mu; bl_cov() -> Sigma + M with M = tau*Sigma - tau*Sigma*P' (P tau Sigma P' + Omega)^-1 P tau Sigma (He-Litterman posterior); bl_weights(risk_aversion) -> unconstrained (delta*Sigma)^-1 mu normalized (NOT long-only). Helpers: market_implied_prior_returns(market_caps, risk_aversion, cov_matrix, risk_free_rate) and market_implied_risk_aversion(market_prices, frequency=252, risk_free_rate).
- PyPortfolioOpt's idzorek_method is a CLOSED FORM, not the paper's least-squares: omega_k = tau * ((1-c)/c) * p_k Sigma p_k' (source verified line ~380-410 of black_litterman.py); conf==0 hardcodes omega=1e6, conf outside [0,1] raises ValueError. The pi, Q, risk_aversion parameters in its signature are UNUSED. For absolute views under unconstrained MVO this closed form is mathematically exact (the tilt scales as tau*v/(tau*v + omega) = c, so tilt = c * 100%-tilt, which is precisely Idzorek's target), so it is faithful for this 3-asset absolute-view pipeline.
- Because omega_k = tau*((1-c)/c)*v_k with v_k = p_k Sigma p_k', the inverse map c_k = tau*v_k / (tau*v_k + omega_k) is EXACT: setting omega directly to the WorldModel's forecast variance sigma_model,k^2 is equivalent to passing view_confidences c_k = tau*v_k/(tau*v_k + sigma_model,k^2) with omega='idzorek'. The pipeline can therefore keep the Idzorek API and still preserve the model's genuine uncertainty losslessly (for absolute views v_k = Sigma_kk).
- Version/packaging state (verified via PyPI JSON 2026-07-07): repo moved from robertmartin8/PyPortfolioOpt to the PyPortfolio org, now maintained by Tuan Tran; latest pyportfolioopt==1.6.0 (uploaded 2026-02-26); deps cvxpy>=1.1.19, numpy>=1.26,<3, pandas<4, scikit-learn>=0.24.1 -- Python 3.12 fine. CURRENT source defaults risk_free_rate=0.0 in market_implied_prior_returns and market_implied_risk_aversion, while readthedocs 'latest' (1.5.4 docs) still documents 0.02 -- live docs/code drift.
- market_implied_prior_returns literally computes risk_aversion * Sigma.dot(w_mkt) + risk_free_rate, i.e. it RE-ADDS rf to produce total returns when rf != 0; views Q must then be in the same convention or the blend is silently biased.
- cvxpy handoff recipe: mu = bl.bl_returns(), S = bl.bl_cov() (both pandas, indexed by cov_matrix tickers); convert to numpy in one canonical asset order; solve maximize(mu@w - (delta/2)*cp.quad_form(w, cp.psd_wrap(S)) - gamma*cp.norm1(w - w_prev)) s.t. [cp.sum(w)==1, lb<=w<=ub]. quad_form requires PSD; BL posterior Sigma+M can be epsilon-non-PSD numerically. With no views the whole chain must return w_mkt (Pi = delta*Sigma*w_mkt inverts exactly, and w_mkt is feasible long-only) -- a free end-to-end anchor test.
- Market-cap sourcing reality for SPY/ACWI/AGG: no free programmatic API returns underlying index market caps (S&P DJI, MSCI factsheets, SIFMA bond outstanding are PDF/manual); yfinance gives only ETF AUM via Ticker.info['totalAssets'] (scrape-fragile, and SPY AUM ~30x ACWI-ETF AUM so an AUM prior is dominated by SPY); Idzorek footnote 4 explicitly sanctions 'a presumed efficient benchmark' as an alternative to true market-cap weights, which legitimizes fixed config benchmark weights.
- Structural universe fact: S&P 500 constituents are a subset of MSCI ACWI (US is roughly 60-65% of ACWI cap), so 'market caps' of SPY and ACWI double-count US large-cap; SPY/ACWI daily-return correlation is ~0.95+, making Sigma ill-conditioned for a 3-asset universe -- BL's own solve is on the well-conditioned KxK matrix (PyPortfolioOpt uses np.linalg.solve with lstsq fallback), but unconstrained bl_weights and an unshrunk MVO will produce extreme SPY-vs-ACWI corner tilts from tiny posterior return differences.

**Landmines**
- ISSUE: Market-cap equilibrium prior is ill-posed for this universe: SPY is contained in ACWI (double-counted US large-cap), and no reliable API serves the underlying index caps; naive choices (ETF AUM, raw index caps) each produce a distorted w_mkt.
  - WHY: w_mkt anchors the whole BL prior Pi = delta*Sigma*w_mkt; a distorted anchor silently biases every posterior and makes the 'no-view => market portfolio' invariant meaningless.
  - ACTION: Plan must fix w_mkt explicitly as a config constant (recommend a documented strategic benchmark, e.g. approx index-cap-derived weights with the SPY/ACWI overlap decision stated, citing Idzorek footnote 4 'presumed efficient benchmark'), stored in one config file with a comment giving source + as-of date, plus a low-frequency manual-refresh operator step. Do NOT scrape market caps in the 5-min loop.
- ISSUE: WorldModel 'uncertainty' semantics: if the JEPA head emits predictive (aleatoric) return variance, it is the same order as Sigma_kk*H, and the implied Idzorek confidence collapses to c ~= tau/(1+tau) ~= 5% for every view, forever -- views never move the portfolio.
  - WHY: BL's Omega is uncertainty in the ESTIMATE OF THE MEAN (that is why tau ~ 0.025-0.05 scales Sigma in the prior); return variance is ~20-40x larger, so misinterpreting the head makes the entire model layer a no-op that looks 'working'.
  - ACTION: Plan must define the WorldModel uncertainty output as epistemic variance of the mean forecast (ensemble spread / MC-dropout variance of mu, or a calibrated head), state the units contract, and add a monitoring assertion: log implied confidences each refresh and warn if all c_k < 0.10 for N consecutive runs (silent-no-op detector).
- ISSUE: Excess-vs-total return convention drift: current pypfopt source defaults risk_free_rate=0.0 while readthedocs still documents 0.02; market_implied_prior_returns re-adds rf; Idzorek's paper requires subtracting rf from total-return views.
  - WHY: A mixed convention (prior in total returns, Q in excess, or vice versa) biases every posterior by rf with no error raised -- classic producer-consumer drift invisible to unit tests (code-quality rule).
  - ACTION: Adopt ONE convention in the plan: everything internal in EXCESS returns; pass risk_free_rate=0.0 explicitly at every call site (never rely on defaults); subtract FRED short rate (e.g. DTB3) from WorldModel total-return forecasts at the single adapter boundary; UI displays total = excess + rf. Add a unit test asserting pi == delta*Sigma*w_mkt exactly.
- ISSUE: Units/horizon mismatch: Sigma is conventionally annualized from daily returns (x252) while the WorldModel emits per-horizon-H returns and variances; PyPortfolioOpt's tau=0.05 and delta defaults assume annualized inputs.
  - WHY: Feeding H-day Q against annualized Sigma scales views by ~252/H, producing wildly overconfident or inert tilts with no error.
  - ACTION: Plan defines a single units contract (annualized decimal excess returns/variances everywhere past the adapter): mu_annual = mu_H * 252/H, sigma2_annual = sigma2_H * 252/H, with ANNUALIZATION=252 and H defined once in a leaf constants module (one-source-of-truth rule) and a round-trip test.
- ISSUE: Confidence/Omega double-encoding: PyPortfolioOpt supports omega='idzorek' (tau cancels) AND direct Omega ndarray (tau does NOT cancel). Mixing modes, or hand-tuning tau while in idzorek mode, silently changes nothing or everything.
  - WHY: tau appears in both the prior term and the closed-form omega; in idzorek mode the ratio is invariant, in direct mode it is load-bearing -- a config that flips modes without revisiting tau produces incomparable runs.
  - ACTION: Plan specifies one enum config bl_omega_mode in {idzorek_confidence, direct_model_variance}, documents the exact equivalence mapping c_k = tau*v_k/(tau*v_k + sigma_model_k^2), pins tau=0.05 as an explicit named constant, and adds a test that both modes produce identical posteriors under the mapping.
- ISSUE: View-order misalignment: view_confidences is positional (view_confidences[view_idx] in source) while absolute_views is a dict; nothing ties confidence k to ticker k.
  - WHY: A reordered dict or a confidence list built from a different iteration order silently attaches SPY's confidence to AGG's view -- wrong posteriors, no exception, invisible to mocked tests.
  - ACTION: Plan mandates one View dataclass list (ticker, q, sigma2/conf) as the single source; P, Q, and view_confidences are all derived from that list in one builder function; add a regression test asserting alignment for a shuffled input.
- ISSUE: Degenerate confidences: conf==0 hardcodes omega=1e6 (an absolute, unit-blind constant) and conf==1 gives omega=0; conf outside [0,1] raises ValueError at runtime.
  - WHY: A model emitting sigma -> 0 (c -> 1) or a percentage passed as 25 instead of 0.25 either crashes the 5-min refresh loop or force-pins the posterior to raw views.
  - ACTION: Clamp c_k to [0.01, 0.99] in the adapter before calling PyPortfolioOpt; validate 0 <= c <= 1 with a descriptive error naming the offending view; unit-test the 0, 1, and 25-instead-of-0.25 cases.
- ISSUE: delta from market_implied_risk_aversion goes negative or near-zero whenever the lookback window's mean excess return is <= 0 (any bear-market window), producing a negative/absurd prior Pi.
  - WHY: delta multiplies Sigma*w_mkt directly; a negative delta flips the sign of every equilibrium return and the optimizer then maximizes into the flip -- garbage weights displayed as advice.
  - ACTION: Plan either fixes delta=2.5 as a constant, or computes it from >=5y of SPY prices and clamps to [1.0, 8.0] with a logged warning when clamped; add a unit test with a synthetic down-market series asserting the clamp engages.
- ISSUE: Ill-conditioned Sigma (SPY/ACWI corr ~0.95+): sample covariance on 3 assets makes bl_weights and an unshrunk cvxpy MVO produce extreme SPY-vs-ACWI corner allocations from basis-point posterior differences.
  - WHY: This is exactly the Best-Grauer input-sensitivity problem BL is meant to fix, reintroduced through the covariance estimate; long-only+box hides it as persistent 100%-corner solutions.
  - ACTION: Plan specifies Ledoit-Wolf shrinkage (pypfopt.risk_models.CovarianceShrinkage(prices).ledoit_wolf()) as THE Sigma estimator; forbids bl_weights on the product path (display-only at most); relies on box constraints per asset; adds an anchor test that a small Q perturbation (10bp) moves weights < some bound.
- ISSUE: BL posterior covariance bl_cov() = Sigma + M can be epsilon-non-PSD after float arithmetic; cvxpy quad_form raises 'Problem does not follow DCP rules' on non-PSD input.
  - WHY: Intermittent hard failure of the refresh loop that depends on data noise -- appears only in production, never in fixture tests.
  - ACTION: At the BL->optimizer adapter: symmetrize S = (S + S.T)/2 and wrap cp.psd_wrap(S) (or clip eigenvalues at 0); assert min eigenvalue > -1e-8 before solve; include a deliberately perturbed near-PSD matrix in tests.
- ISSUE: Two optimizers drift: PyPortfolioOpt's EfficientFrontier also wraps cvxpy; using it for some paths and a hand-rolled cvxpy program for turnover constraints creates dual sources of truth for the optimization.
  - WHY: Objective conventions differ (PyPortfolioOpt max_quadratic_utility uses its own risk_aversion scaling, L2_reg gamma, weight bounds semantics); results diverge silently and debugging compares apples to oranges.
  - ACTION: Plan scopes PyPortfolioOpt to exactly two jobs: black_litterman module + CovarianceShrinkage. ONE hand-rolled cvxpy module owns the mean-variance-utility problem (long-only, sum=1, box, L1 turnover). State this boundary in the plan's architecture section.
- ISSUE: Docs/version skew: readthedocs 'latest' still renders 1.5.4 docs while PyPI ships 1.6.0 from the new PyPortfolio org (maintainer change Feb 2026); defaults (risk_free_rate) already differ between the two.
  - WHY: An agent implementing from the docs will encode 0.02-rf behavior the installed library no longer has; future 1.x releases under the new maintainer may move APIs again.
  - ACTION: Pin pyportfolioopt==1.6.0 in pyproject; add an import-time smoke test constructing BlackLittermanModel(cov, absolute_views=..., omega='idzorek', view_confidences=[...]) on fixture data and asserting a known posterior vector (golden-value test) so any silent API/behavior change fails CI.
- ISSUE: No-view anchor missing: nothing in the stated pipeline verifies that with views disabled (or all-zero confidence) the system reproduces w_mkt.
  - WHY: This is the one free, exact invariant of BL (Pi inverts to w_mkt; w_mkt is long-only feasible); without it, unit/scale/convention bugs anywhere in the chain are undetectable (measurement-validity: calibrate with anchors).
  - ACTION: Plan includes an integration anchor test through the production pipeline entry point: zero-confidence (or K=0) views + turnover term disabled must return w ~= w_mkt within 1e-6, and a known-garbage anchor (absurd Q with c=0.99) must move weights materially. Run both in CI on fixture prices.

**Recommendations**
- Concrete BL recipe for the plan: (1) w_mkt = config constant (documented benchmark, Idzorek fn.4); (2) Sigma = CovarianceShrinkage(prices, frequency=252).ledoit_wolf() on daily returns; (3) delta = fixed 2.5 or clamped market_implied_risk_aversion(5y SPY, risk_free_rate=fred_rf); (4) pi = market_implied_prior_returns(w_mkt-as-caps, delta, Sigma, risk_free_rate=0.0) i.e. pure excess delta*Sigma*w_mkt; (5) views = one ordered list of View(ticker, mu_excess_annualized, sigma2_epistemic_annualized); (6) BlackLittermanModel(Sigma, pi=pi, absolute_views={t: q}, omega='idzorek', view_confidences=[c_k], tau=0.05) with c_k = clamp(tau*Sigma_kk/(tau*Sigma_kk + sigma2_k), 0.01, 0.99); (7) mu_post = bl.bl_returns(), S_post = bl.bl_cov(); (8) hand-rolled cvxpy: maximize(mu_post@w - delta/2*quad_form(w, psd_wrap(S_post)) - gamma_tc*norm1(w - w_prev)), constraints [sum(w)==1, 0<=w<=box].
- Use the exact inverse mapping to honor both the spec and the model: primary mode passes Idzorek confidences derived from model variance via c_k = tau*v_k/(tau*v_k + sigma_model_k^2) (lossless under PyPortfolioOpt's closed form); the UI card for the BL stage displays per-view: Q_k vs prior pi_k, c_k, omega_k, and the per-asset tilt w - w_mkt (the paper's Table 7 'implied confidence' framing makes an excellent one-card visualization).
- Run the entire internal pipeline in annualized EXCESS returns; rf comes from FRED (DTB3/DGS3MO) already ingested for the macro layer; convert WorldModel horizon-H total-return forecasts at exactly one adapter module (subtract rf, annualize mean by 252/H and variance by 252/H); display-layer adds rf back.
- Structure the BL stage as a pure function bl_blend(sigma, w_mkt, delta, views, tau, mode) -> (mu_post, S_post, diagnostics) with diagnostics = {pi, per-view omega, implied confidence, tilt vector}; this makes the anchor tests trivial and gives the React card its payload for free.
- Do not use bl.bl_weights()/optimize() in the product path (unconstrained, can go short, ignores turnover); it is acceptable only as a debug diagnostic on the card.
- Pin pyportfolioopt==1.6.0 (new PyPortfolio-org release, Feb 2026, numpy>=1.26/py3.12-compatible) and add the golden-value smoke test; keep cvxpy pinned to a version tested with both the hand-rolled optimizer and pypfopt's cvxpy>=1.1.19 floor.
- Write the Idzorek paper's 8-asset worked example (Tables 1-7 values are all in the local PDF extraction) as a fixture test of the BL module: reproduce E[R] and w-hat from Table 6 within tolerance using the He-Litterman omega, and the implied confidences of Table 7 -- this validates the implementation against the primary source, not a re-derivation.

**Open decisions**
- w_mkt sourcing: fixed documented benchmark weights (recommended) vs ETF-AUM-derived vs underlying-index caps -- and if index caps, how to treat the SPY-inside-ACWI overlap (subtract S&P 500 cap from ACWI's, or accept double-counting).

- Units convention statement: annualized-excess everywhere internal (recommended) vs horizon-H units end-to-end.

- ISSUE: Daily FRED series have three incompatible calendars: DFF publishes 7 days/week (weekend rows), DGS10/T10Y2Y skip and/or blank-string market holidays (T10Y2Y 2026-07-03 is an empty value -> NaN), VIXCLS has its own holiday pattern. A naive join across them and the NYSE price calendar drops rows or propagates NaNs into features.
  - WHY: Verified live in fredgraph.csv output. An inner join on raw dates silently shrinks the training set; an outer join injects NaNs that PyTorch will happily turn into NaN losses mid-training.
  - ACTION: Declare the canonical calendar = the trading dates present in the SQLite price table. Reindex every macro series to that calendar via as-of backward fill (after the release-lag shift), explicitly parsing FRED empty strings ('.') as missing first. Add one ingestion test asserting no NaNs and no weekend dates in the joined feature frame.
- ISSUE: FRED serves latest-REVISED history: CPIAUCSL/UNRATE/PAYEMS historical values in a fresh pull differ from what was known at the time (benchmark revisions, seasonal-factor updates). Training the world model on revised macro inflates apparent predictive power beyond the lookahead issue above.
  - WHY: This is distinct from release lag — even with correct available_date joins, the VALUE at each historical date is today's revision, not the vintage. For an advisory tool this may be acceptable, but it must be a decision, not an accident.
  - ACTION: Plan must contain an explicit V1 decision: 'V1 accepts revised-history bias (documented limitation)' OR 'V1 pulls vintages via fredapi.get_series_all_releases'. Either way, make the macro table append-only with ingested_at so the DB becomes genuinely point-in-time from first deploy forward.
- ISSUE: Google Finance 'beta scrape tertiary' is not a real source: no official API since 2012, GOOGLEFINANCE is Sheets-only, and all 2026 'Google Finance APIs' are paid third-party scrapers; direct HTML scraping fights consent pages/anti-bot and violates ToS.
  - WHY: Any plan step that says 'best-effort Google Finance scraper' will either burn a build-step producing brittle throwaway code or quietly ship dead code — and it adds zero resilience because it fails under the same conditions (anti-bot walls) that just killed Stooq.
  - ACTION: Cut Google Finance from V1 entirely. The tertiary tier becomes the SQLite cache + staleness banner (see landmine 1). If a third LIVE source is a hard requirement, spec a keyed free-tier provider (Tiingo/Alpha Vantage) instead — cheaper to build than a scraper and durable.
- ISSUE: Cross-source adjusted-price inconsistency: yfinance auto-adjusted closes (splits+dividends) will not match a fallback source's adjustment policy (raw closes, split-only, or dividend-adjusted-differently), producing artificial return spikes exactly on the days the fallback engages.
  - WHY: The AFML feature layer computes returns/vol from the price series; a source switch mid-series injects a fake -1.5%/+1.5% 'return' the model and optimizer will treat as signal. Classic producer-consumer drift invisible to unit tests (code-quality.md).
  - ACTION: Schema records source per (ticker, date) row and defines ONE canonical adjustment policy. On any source failover, require an overlap reconciliation: fetch N overlapping days from the new source, assert max relative close deviation < tolerance (e.g. 0.1%) before appending; on breach, halt ingestion for that ticker with a stable error and keep serving cache. Put this in the ingest step's acceptance criteria.
- ISSUE: History-length mismatch caps the joint training window: ACWI's inception is 2008-03-28, so SPY/ACWI/AGG only co-exist for ~18.3 years (~4,600 daily observations) — small for training even a minimal JEPA, and the window starts mid-financial-crisis.
  - WHY: Verified via yfinance period='max'. A plan that assumes 'decades of daily data' will over-scope the model; 4.6k samples with heavy temporal autocorrelation is few-shot territory for deep nets and invites overfitting the 2008 regime at the window edge.
  - ACTION: Plan states training window = intersection (2008-03-28 onward) explicitly, sizes the JEPA accordingly (small, heavy regularization, purged/embargoed CV per AFML), and records the alternative (per-asset masking or index-proxy backfill, e.g. splicing ACWI with the MSCI ACWI index) as a decided-against or later option.
- ISSUE: Timezone/date-key mismatch: yfinance returns tz-aware timestamps (America/New_York) while FRED dates are naive calendar dates; SQLite has no native datetime type. Joins on mixed keys silently misalign by one day or fail equality.
  - WHY: A one-day misalignment between price returns and daily macro (e.g. VIXCLS) is undetectable by tests that only check row counts, and shifts every feature's information content.
  - ACTION: Normalize at the ingestion boundary in ONE place: strip tz, store trading date as TEXT ISO-8601 (YYYY-MM-DD) everywhere, and add a schema test asserting the date-key format. Name this the single source of truth for the date key (code-quality.md one-source-of-truth).
- ISSUE: FRED_API_KEY handling: fredapi needs the key; the plan stores it in a gitignored .env. If the key is missing/invalid the naive failure is a stack trace deep in the refresh loop, or silent empty macro features.
  - WHY: security.md: 'don't do X until Y' sentences must become startup invariants. Silent empty macro features would degrade the model invisibly — worse than crashing (measurement-validity fail-loud principle applies to ingestion too).
  - ACTION: Startup check: FRED_API_KEY present AND one live probe call succeeds, else the app starts in explicit 'macro-disabled' degraded mode with a stable error code shown on the macro ingest card — never silently-empty features. Optionally wire the keyless fredgraph.csv endpoint as the degraded-mode fetch path.

**Recommendations**
- Source hierarchy for V1: yfinance (>=1.5, pinned) PRIMARY at once-daily incremental fetch; SQLite last-known-good cache as the guaranteed fallback with UI staleness surfacing; optional keyed Tiingo or Alpha Vantage adapter as the only realistic second LIVE source. Drop Stooq (account/API-key gated + PoW wall since ~March 2026) and Google Finance (no API exists) from V1 scope.
- Wrap all price sources behind one SourceAdapter interface (fetch_daily(ticker, start, end) -> canonical frame) with per-adapter health status rendered on the ingest pipeline card — this makes the 'which source served today's data' question first-class in the UI the plan already wants.
- Verified V1 macro set with frequencies: DGS10 (daily, business days), T10Y2Y (daily, blank on market holidays), VIXCLS (daily), DFF (daily incl. weekends), BAMLH0A0HYM2 HY OAS (daily), DTWEXBGS broad dollar (daily, ~3-day lag), NFCI (weekly, Friday-dated, released following Wednesday), CPIAUCSL (monthly, ~40-day availability lag), UNRATE (monthly, ~2-3 day lag after month end), PAYEMS (monthly, same release as UNRATE). All IDs verified live against fredgraph.csv on 2026-07-07.
- Macro-to-daily join recipe for the plan: (1) parse FRED '.'/empty as NaN; (2) shift each series by its declared release lag to get available_date; (3) merge_asof backward onto the NYSE trading-date spine derived from the price table; (4) assert no NaNs/weekends in the output. Ship this as one tested function — it is the highest-bug-density 30 lines in the whole data layer.
- Make the refresh semantics explicit in the plan: '5-minute refresh' = pipeline recompute from SQLite; data fetch = scheduled daily post-close + on-demand with min-interval guard and 429 backoff. This one sentence prevents both the Yahoo ban and the FRED quota concern.
- Add a tiny ingestion smoke test that runs against the real network once (not in CI): fetch 5 days of SPY via the adapter, fetch DGS10 via fredapi, assert row shapes — per measurement-validity, this catches the next yfinance/Yahoo breakage window at ingest time instead of as garbage model output.

**Open decisions**
- Fallback tier: cache-only (zero new deps) vs adding one keyed free-tier provider (Tiingo or Alpha Vantage) — and if keyed, which provider and where its key lives (.env alongside FRED_API_KEY with the same startup probe).
- Whether to attempt Stooq at all as an operator-setup-gated adapter (create account, obtain API key, undocumented endpoint mechanics) — recommended NO for V1.
- Canonical modeling price: auto-adjusted close (total-return-like, recommended for return features) vs raw close — must be stated in the schema before the feature layer is specced.
- Point-in-time strictness for V1: fixed per-series release-lag constants (cheap, ~day-level accurate) vs exact ALFRED vintages via fredapi get_series_all_releases (correct, more code + slower backfill).
- Training window policy given ACWI's 2008-03-28 inception: intersection window (2008->, ~4,600 obs) vs per-asset masking vs splicing pre-2008 index proxies — affects JEPA architecture sizing.
- Whether monthly macro (CPIAUCSL/UNRATE/PAYEMS) enters the V1 feature set at all, given horizon H is short and their information updates ~monthly with multi-week lags — a daily-series-only V1 macro set (DGS10, T10Y2Y, VIXCLS, DFF, HY OAS, dollar) sidesteps most of the point-in-time machinery and could be the V1 cut-line.
